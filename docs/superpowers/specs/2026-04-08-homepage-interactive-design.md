# 博客首页动态交互改造设计

- 日期：2026-04-08
- 站点：`/Users/zhangyukun/project/hugh-note`
- 目标：首页从文章列表改为数据驱动的动态入口页，原文章流迁移到 `/blog`

## 1. 背景与目标

当前首页 `/` 仍是 `archie` 主题默认的文章列表页。站点已经包含三类核心内容：

- 博客正文：`/posts`
- 疯言疯语：`/crazy-talk/`
- 跑步页：`/running/`

新首页需要参考 `https://loongphy.com/` 的“动态入口页”思路，但不能脱离当前博客的视觉语言。首页应该优先展示“持续输出”和“持续跑步”两个主题，而不是继续承担文章归档页职责。

目标如下：

- `/` 变为新的动态首页
- 原首页文章流迁移到 `/blog`
- 首页主角是两个热力图：博客更新热力图、跑步热力图
- 首页包含疯言疯语动态展示区
- 首页导航链接替换为用户自己的链接
- 保持当前 Hugo + `archie` 体系，避免做成独立 SPA

## 2. 需求结论

已确认的产品决策：

- 首页采用“数据先行”布局，两个热力图同时出现在首屏
- `/blog` 承接原首页文章列表
- 跑步热力图按“当月”展示，目标固定为每月 `150 km`
- 博客更新热力图按“当天 `content/` 是否有内容更新”统计，不设置目标
- 博客更新口径只看 `content/`，不看仓库内其他目录变更
- 疯言疯语首页展示方式为动态句子区，轮播或随机展示多条内容
- 首页保留 `Blog / 疯言疯语 / Running` 三个显著入口

站点链接：

- `Blog`: `https://hugh-zhan9.github.io/`
- `疯言疯语`: `https://hugh-zhan9.github.io/crazy-talk/`
- `Running`: `https://hugh-zhan9.github.io/running/`

## 3. 信息架构

调整后的页面职责：

- `/`
  - 新的动态首页
  - 展示博客更新热力图、跑步月目标热力图、疯言疯语动态区
- `/blog`
  - 原首页文章流
  - 保持现有文章列表行为
- `/crazy-talk/`
  - 保持现有疯言疯语列表页
- `/running/`
  - 保持现有跑步页，不改其项目结构

首页页面层级：

1. 顶部导航区
   - 站点标题
   - `Blog / 疯言疯语 / Running` 三个入口
2. 首屏主区
   - 博客更新热力图模块
   - 跑步月目标热力图模块
3. 下部辅助区
   - 疯言疯语动态句子区
   - `/blog` 明确入口

## 4. 视觉设计

视觉方向：延续现有博客的克制、极简、偏代码感的排版基础，但让首页更像“封面面板”而不是列表页。

保留项：

- 站点整体的极简气质
- 现有字体体系和基础排版节奏
- Hugo 静态站点的结构感

增强项：

- 首页使用独立的深色面板化布局
- 采用浅色正文 + 暖色局部强调，而不是继续沿用默认紫色变量
- 两个热力图使用统一的格子语言，但强调层次不同
- 加入少量入场动画、hover 反馈、轮播切换

明确不做：

- 不照搬参考站配色或组件细节
- 不引入复杂滚屏叙事
- 不改造整站为重前端应用

## 5. 数据设计

### 5.1 博客更新热力图

数据来源：当前博客仓库 `content/` 下的内容文件。

统计口径：

- 只要“`content/` 下当天有内容更新”就算一次更新日
- 统计范围只覆盖 `content/` 下的博客内容
- 同一天有多次内容变更时，可以用于映射不同热力等级

展示目标：

- 不显示“目标值”
- 强调持续更新、最近活跃度和更新时间分布

展示维度：

- 以近一段时间的日历格热力图展示
- hover 显示日期和更新强度

实现建议：

- 构建阶段从 `content/` 内容文件中提取按日聚合的更新分布
- 首页模板直接消费处理后的结构化数据

### 5.2 跑步热力图

数据来源：现有 `/running/` 页面公开静态资源中的活动数据。

已验证事实：

- `/running/` 是静态部署的前端应用
- 其资源中包含完整活动数组
- 数据可按日期聚合为每日公里数

统计口径：

- 仅展示当前月份
- 每日格子表示当天跑量强度
- 显示当月累计公里数与目标 `150 km` 的完成进度

展示内容：

- 当月标题，例如 `2026 年 4 月`
- 已跑公里 / 目标公里
- 进度条
- 当月每日热力格
- hover 显示具体日期和公里数

实现建议：

- 构建阶段读取 `/running/` 页面相关静态资源
- 提取跑步类型数据
- 按当前月份聚合出每日公里数和累计里程

### 5.3 疯言疯语动态区

数据来源：`content/crazy-talk/`。

展示方式：

- 首页展示多条疯言疯语
- 自动轮播或随机切换
- 允许手动切换到下一条

展示目标：

- 提供页面中的轻动态文本层
- 与两个热力图形成“数据 + 句子”的节奏对比

## 6. 交互设计

首页交互保持轻量，核心是“可感知”而不是“炫技”。

交互清单：

- 页面首次加载时整体淡入 + 模块分段进入
- 三个入口链接 hover 时有轻微位移和底色反馈
- 博客热力图格子 hover 显示日期和活跃信息
- 跑步热力图格子 hover 显示日期和当天公里数
- 疯言疯语区自动轮播
- 疯言疯语区支持手动切换

不做：

- 不做复杂状态机
- 不做全页滚动动画
- 不做依赖大型前端库的交互系统

## 7. 技术方案

推荐方案：使用 Hugo 原生首页改造。

原因：

- 与当前站点技术栈一致
- 部署方式不变
- 长期维护成本最低
- 能满足所需的轻交互与数据展示

预计改动范围：

- 新增站点级首页模板，覆盖主题默认首页
- 新增 `/blog` 列表页模板
- 新增首页专用样式文件
- 新增首页专用脚本文件
- 在 Hugo 配置中补充首页所需参数或资源引用

实现边界：

- 不引入大型 JS 框架
- 不修改 `/running/` 项目本身
- 不打破现有 Hugo 内容组织方式

## 8. 模块拆分

建议的首页模块：

- `Hero/Nav` 模块
  - 展示站点标题和三个外部/内部入口
- `Blog Heatmap` 模块
  - 展示博客更新热力图
- `Running Heatmap` 模块
  - 展示当月跑量与目标进度
- `Crazy Talk Ticker` 模块
  - 展示疯言疯语动态句子
- `Blog CTA` 模块
  - 链接到 `/blog`

数据处理职责：

- 博客热力数据处理
- 跑步数据抓取与当月聚合
- 疯言疯语列表提取与轮播初始化

## 9. 数据流

构建阶段：

1. Hugo 读取站点内容
2. 首页模板或辅助处理逻辑收集 `content/` 更新日期数据
3. 首页模板或辅助处理逻辑读取跑步页面静态资源或运行时拉取跑步数据
4. 生成首页需要的博客热力数据、疯言疯语列表，并为跑步模块准备数据入口
5. 输出静态 HTML、CSS、JS

运行阶段：

1. 首页加载静态页面
2. JS 初始化轻交互
3. 博客热力图直接使用构建时输出的数据
4. 跑步模块在页面加载时请求最新跑步数据并更新当月目标区
5. 疯言疯语轮播开始工作
6. 热力图 hover 展示 tooltip 或 title 信息

## 9.1 数据刷新策略

首页本身仍然是静态页面，但数据刷新采用“静态壳 + 分层刷新”的方式：

- 博客热力图：构建时生成
- 疯言疯语动态区：构建时生成
- 跑步热力图：运行时拉取最新跑步数据

具体刷新行为：

- 当 `content/` 内容更新并重新部署博客时，博客热力图和疯言疯语内容随新构建一起更新
- 跑步热力图不依赖博客重新部署，首页每次打开时都请求一次最新跑步数据
- 若需要，也可以加一个短时缓存策略，例如同一会话内不重复请求

这样可以保证：

- 首页仍是 Hugo 静态页面，结构简单
- 博客相关内容和博客部署节奏一致
- 跑步数据能跟着 `/running/` 的最新数据刷新，而不需要重新发布博客首页

## 10. 错误处理与降级

博客热力图：

- 若某些内容文件缺失预期日期信息，按可获取到的日期继续渲染
- 若统计强度失败，至少保留“是否更新”的二值热力图

跑步热力图：

- 若运行时请求 `/running/` 数据失败，首页显示降级文案
- 若当前月份无跑步数据，仍显示 `0 / 150 km` 与空热力图
- 若跑步数据结构变化，避免整个首页渲染失败，改为显示模块级空状态

疯言疯语：

- 若条目不足，允许单条循环
- 若无内容，显示静态占位文案并保留入口链接

## 11. 响应式设计

桌面端：

- 顶部导航横向展开
- 两个热力图模块并排展示
- 疯言疯语位于下方横向区域

移动端：

- 模块按垂直顺序堆叠
- 先导航，再博客热力图，再跑步热力图，再疯言疯语
- 保持点击区域充足，避免热力格过小难以触达

## 12. 测试策略

需要覆盖的验证点：

- `/` 能正确渲染新的动态首页
- `/blog` 能正确渲染原文章列表
- `Blog / 疯言疯语 / Running` 链接正确
- 博客热力图能生成有效格子数据
- 跑步热力图能显示当月公里数和 `150 km` 目标
- 跑步数据缺失或抓取失败时有降级状态
- 疯言疯语轮播在无 JS 致命错误的情况下可运行
- 桌面端和移动端布局都可用

建议验证方式：

- 本地 Hugo 构建与页面检查
- 针对数据处理逻辑增加脚本级验证
- 对关键路由与关键文案做契约测试

## 13. 风险与取舍

主要风险：

- `/running/` 的静态资源文件名带 hash，后续发布后可能变化
- 运行页数据结构如果调整，首页运行时解析逻辑可能失效
- Hugo 模板中直接做复杂数据加工会影响可读性

取舍方案：

- 尽量将解析逻辑收敛为清晰的小块，而不是堆在单个模板中
- 对跑步数据运行时解析失败提供清晰降级
- 首页交互保持轻量，减少前端复杂度

建议按以下顺序实施：

1. 建立 `/blog`，迁移原首页列表
2. 搭建新首页 HTML 结构
3. 接入博客热力图数据
4. 接入跑步月目标数据
5. 接入疯言疯语动态区
6. 补样式与轻交互
7. 做响应式与降级检查
8. 完成测试与回归
